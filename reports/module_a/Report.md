### Модуль А

### Цель
Разработка системы транспортной аналитики на участке дороги с записи с видеокамеры.
Реализация потокового и пакетного контура обработки
Классификация т/с
Представление витрин в удобном UI дашборде.
Realtime обновление данных

### Функциональные и нефункциональные требования:
#### Функциональные: 
1) Детекция и классификация т/c, трекинг между кадрами
2) Потоковая и пакетная обработка
3) Запись кадров детекции в хранилище
4) Формирование предсказаний на основе исторических данных
5) Оценка качества данных (DQ)
#### Нефункциональные
1) Realtime обновление витрин
2) Прогнозирование на 30/60/120 минут вперед
3) Добавление новых полей / источников данных происходит без перестройки архитектуры
4) Единая версия правды
5) Хранение истории измененений за последние 180 дней


### Стек и его обоснование
БД: ClickHouse - колоночная СУБД, предзназначенная для аналитике большого кол-во данных.
Поддерживает партиционирование, движок Merge Tree - быстрая вставка и работа с данными

Модель для детекции и классификации: YOLO - Предобученная модель на датасете COCO, 
встроенные object_type для транспортной аналитики, встроенный трекинг объектов между кадрами.

Модель для прогнозирования: LightGBM - Технология градиентного бустинга, быстро обучается на табличных данных. 
После обучения фиксирует средние абсолютные ошибки (MAE), подходит для предиктивной аналитике на 30/60/120 мин.

Оркестратор данных для потоковой и пакетной обработки: Apache Kafka - Opensource инструмент, Producer и Consumer

Задачи по расписанию для подсчета метрик с заданной временной гранулярностью (DAG): Apache Airflow - Отлично подходит для вызова задач по расписанию / с заданным интервалом, позволяет реализовывать порядок вызова задач (функций), что хорошо подходит для реализации ETL процесса.

Аналитический Дашборд (Витрины): Apache Superset - удобный и понятный пользователю интерфейс, популярный инструмент для построение графиков / диаграмм на основе данных из БД.

Хранилище для кадров с видеоролика: MinIO - S3 типное хранилище, используется для хранения аватарок / файлов в проектах, создает удобную структуру хранения данных, которая отлично подходит для данной задачи date/camera_id/frame_n.jpg

Дэшборд для контроля качества данных и состояния системы: Grafana - Популярное решение для поставленной задачи, связанной с метриками качества.

ЯП: Python - Высокоуровневый язык программирования, набирающий популярность в реализации бизнес решений каждый год,
позволяет удобно работать с вышеперичисленными инструментами, т.к все они поддерживают документацию на данном языке программирования.

### Архитектурное построение DWH (Data Ware House)
Была выбрана 3-ех слояная медальонная архитектура построения DWH: Bronze -> Silver -> Gold,
Которая представлена в виде 3-ех хранилищ, называемых "слоями": raw_layer -> staging_layer -> mart_layer
raw_layer - Сырой слой, в него попадают все данные (ничто не теряется). Как растаки этот слой отвечает за сбор всех данных c детекций
staging_layer - Слой подготовки, в этом слое данные из raw_layer проходят фильтрацию и преобразования для mart_layer
mart_layer - Слой витрин, содержит в себе таблицы, которые отвечают на бизнес вопросы, данные из staging преобразуются под каждую таблицу.

Почему именно медальонная архитектура: Данные с YOLO детекций содержат в себе уверенность модели в детекции. В услови ТЗ сказано, что никакие данные не должны теряться, а темболее исчезать. 
Чтобы выводить наиболее чистые данные в витрины необходимо фильтровать детекции с YOLO, например, где уверенность модели в распозновании > коэфицента распознования, а также обогащать эти данные.

### Набор витрин
Спроектированные витрины и типы данных полей находятся в module_a/draw.io/Module_A.draw.io

### Описание витрин
Витрина realtime_traffic: Витрина, предзначзначенная для формирования realtime метрик
Обновление: Каждые 5 минут
Источник данных: staging_layer.detections_clean
Обоснование интервала обновления: 5 минут - идеальное время для формирования realtime метрик, учитывая только половину времени из этого т/с стоят на сфетоворов.
Поэтому, среднее кол-во уникальных детекций, которое будет обнаружено на кадре ~ 50 - 90.

Витрина hourly_traffic: Витрина, предназначенная для формирования часовых метрик каждый час.
Обновление: Каждый час
Источник данных: realtime_traffic

Витрина systemic_apperances: Витрина, предназначенная для ответа на бизнес вопрос: "Есть ли систематические появляния большегрузного транспорта в часы пик, требующие ограничения или перераспределения потоков?"
Обновление: Каждый час
Источник данных: hourly_traffic

Витрина proximity_events:  Витрина, предназначенная для формирования данных, связанных с расстоянием между двумя т/c, отвечает на бизнес вопрос: "В какие моменты фиксируется опасное сближение т/c"
Обновление: Каждые 5 мин
Источник: staging_layer.detections_clean

Витрина dispatcher_predictions:  Витрина, предназначенная для предективной аналитики данных, отвечает на вопрос: "Какие перспективные рекомендации выдать диспетчеру, чтобы предотвратить затор на горизонте 30-120мин?" 
Обновление: Каждые 120 мин
Источник: DAG (predictions) LightGBM (realtime_traffic, systemic_apperances, proximity_events)
Обоснование частоты обновления: В системе используется технология Градиентного бустинга, каждые 120 мин формируются пресказания на следующие 30/60/120 мин.
Время частоты обновления таблицы обусловлено качеством работы предективной аналитики (чем больше предоставить данных основывая на прошлом, тем точнее предсказание на будущее)

### Не витринные таблицы:
raw_layer.detections_raw: В эту таблицу фиксируются все детекции, которое возвращает модель.
Источник данных: video

staging_layer.detections_clean: В этой таблице формируются чистые детеции и расчитыватся такие показатели как: скорость в час, центры т/c.
В таблицу не попадают детекции, уверенность модели в которых < установленного порога уверенности
Источник данных: raw_layer.detections_raw
